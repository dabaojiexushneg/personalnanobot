# 已知问题与架构取舍

这份文档用于面试时主动说明“我知道系统哪里还不完美，以及为什么现在先这样做”。

## 当前取舍

### 1. 控制面仍然使用 SQLite

现状：

- 对本地开发和面试展示足够
- 对单机控制面友好

取舍：

- 还没有升级到 PostgreSQL / MySQL
- 多实例控制面扩展能力有限

后续方向：

- 把控制面数据库迁到 PostgreSQL

### 2. 任务中心还不是完整工作流引擎

现状：

- 已支持 manual / interval / cron
- 已支持条件流、重试、阻塞原因展示

取舍：

- 还没有可视化多步骤编排
- 还没有审批、分支和更复杂依赖

后续方向：

- 向完整 workflow 能力扩展

### 3. 渠道同步发送仍依赖已有会话目标

现状：

- QQ / 微信同步发送需要已有绑定会话或目标账号

取舍：

- 这是为了避免系统在没有真实上下文时盲发消息

后续方向：

- 支持显式填写目标账号 ID / chat_id

### 4. RAG 还没有更高级检索链路

现状：

- 已有 Milvus 检索、引用和 Trace 命中证据

取舍：

- 还没有 reranker
- 还没有文档版本管理
- 还没有更精细的 chunk 策略和引用去重

后续方向：

- 增加 reranker 与文档版本化

### 5. 前端控制台仍然偏“工程控制台”而不是产品化 UI

现状：

- 控制台强调信息密度和可观测性

取舍：

- 更偏管理端风格
- 用户体验可以继续做得更精致

后续方向：

- 增强筛选、仪表盘、引导与结果展示

## 面试时推荐的说法

可以主动说：

> 这个项目我刻意先把工程主链路打通，包括多助手、RAG、任务、Trace、渠道和安全。当前还保留了一些明确的取舍，比如控制面仍是 SQLite、任务中心还不是完整工作流引擎，但这些都是我清楚知道并规划过下一步的部分。
