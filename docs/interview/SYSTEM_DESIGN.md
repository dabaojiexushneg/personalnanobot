# 系统设计答辩

## 1. 项目目标

`nanobot个人生活账号助手` 的目标不是做一个“会聊天”的单模型壳，而是做一个可长期运行的个人生活账号 AI 应用工作台：

- 多助手隔离
- 多入口接入
- 可插拔工具与 MCP
- 可解释的 RAG
- 任务调度与主动执行
- Trace / Token / 状态观测

## 2. 为什么是多助手架构

一个 AI 应用通常会遇到两个矛盾：

- 所有能力放在一个助手里，提示词和工具容易失控
- 每个能力拆成多个服务，又会让使用体验割裂

这里选择“多助手 + 单控制台”的原因是：

- 每个助手有独立 persona、模型、工具、MCP、工作区
- 对用户来说仍然是一个统一工作台
- 对工程来说更容易做权限边界、Prompt 版本化和问题定位

## 3. 存储边界为什么这样拆

### 控制面：SQLite

放在 SQLite 的内容：

- 用户与鉴权
- 任务调度
- Trace / 事件
- 会话与助手记忆
- Token 统计

原因：

- 结构化、事务性强
- 本地开发成本低
- 适合单机控制面

### RAG：Milvus

放在 Milvus 的内容：

- 文档元数据
- chunk 文本
- 向量索引
- 检索结果统计

原因：

- 向量检索是核心能力
- 需要更明确的 RAG 真相源
- 比 SQLite/Chroma 方案更适合面试里讲“工程化升级”

## 4. 为什么要统一并发语义

AI 应用里最容易出问题的是“会话并发”：

- 同一会话同时进来多条消息，容易串上下文
- 不同渠道 / Web / API 并发语义不一致，容易出现“这里串行、那里竞争”的隐患

当前收敛原则是：

- 同 session 串行
- 不同 session 并发
- Web / API / 渠道入口尽量统一

这样做的原因：

- 保证上下文一致性
- 让 Trace 和 Token 统计更稳定
- 后续扩容时更容易做 worker 化

## 5. 为什么任务中心做成条件流

定时任务如果只是“到时间就跑”，在 AI 应用里很快会暴露问题：

- RAG 未连接还在跑知识摘要
- 渠道离线还在做主动推送
- 上一次刚成功，这一次又重复执行

所以任务中心支持：

- `RAG 已连接`
- `渠道在线`
- `距离上次成功至少 N 分钟`

这类条件的意义是把任务中心从“定时 prompt 执行器”提升成“有条件约束的 AI 工作流入口”。

## 6. 为什么要做 Trace / 引用 / Token 观测

AI 应用如果没有观测，问题通常没法定位：

- 是检索错了，还是模型回答错了
- 是工具调用失败，还是渠道发送失败
- 是成本异常，还是请求量上去了

这里做了三层观测：

- Trace：输入、输出、工具、RAG 证据、错误
- 引用：让回答可解释
- Token：让成本和配额可治理

## 7. 安全设计的取舍

当前默认策略是“开发方便，但生产优先”：

- 默认启用鉴权
- 默认启用 CSRF
- 默认 `Secure + HttpOnly + SameSite`
- 默认不提供固定管理员密码
- 首次启动必须初始化管理员

但同时允许：

- 显式 `devMode=true`
- 本机开发放宽 Cookie Secure

这是因为项目主要面向本地开发和面试演示，同时又要避免安全配置显得过于随意。

## 8. 后续可以继续演进的方向

- 控制面 SQLite 升级到 PostgreSQL
- 任务中心继续往完整 workflow 演进
- RAG 增加 reranker、文档版本管理、引用去重
- Trace 接入更完整的筛选与分析
- 进一步把运行时拆成更细的模块边界
