# 面试演示脚本

这份脚本面向 “AI 应用工程开发” 面试，目标是在 5 到 8 分钟内完整展示项目价值。

## 演示目标

向面试官证明这不是一个单轮聊天 Demo，而是一套具备：

- 多入口接入
- 多助手编排
- Milvus RAG
- Trace 与 Token 观测
- 自动化任务
- MCP/Skills 扩展
- 基础工程化安全与部署能力

的可交付 AI 应用系统。

## 建议演示顺序

### 1. 启动系统

先展示一键启动脚本：

```powershell
.\start-all.ps1 -SkipMilvus
```

如果本机 Milvus 已经运行，也可以直接使用：

```powershell
.\start-all.ps1
```

要点：

- 项目支持 WSL2 + Docker Desktop + Milvus 的本地开发链路
- Web 控制台、OpenAI-style API、渠道服务有统一启动脚本
- 控制台默认要求登录，不是裸奔后台

### 2. 展示控制台总体结构

登录控制台后，先快速扫 4 个区域：

- 助手集群
- 主对话工作台
- 知识库 / RAG
- 任务中心 / Trace

建议一句话说明：

> 这个控制台把 AI 应用最常见的几条链路放在一处统一管理：多助手、多模型、RAG、任务、渠道和可观测性。

### 3. 演示 Milvus RAG

步骤：

1. 上传一份知识文档
2. 在 “知识库 / RAG” 里看到文档分块与状态
3. 在主对话工作台提一个知识相关问题
4. 展示回答后面的来源引用卡片
5. 打开 Trace 详情，展示 `knowledge_hit` 证据

推荐说明：

- 控制面状态在 SQLite
- RAG 唯一真相源在 Milvus
- 回答不仅返回文本，还返回引用证据和 Trace 命中记录

### 4. 演示多模型分工

输入一条图像生成请求，例如：

```text
生成一张 AI 助手产品宣传图
```

要点：

- 文本理解由 `qwen3.5-plus` 完成
- 生图通过 `generate_image` 工具切到 `Doubao-Seedream-4.0`
- 说明这是“主模型 + 专用工具模型”的典型 AI 应用编排

### 5. 演示 Trace 与 Token 观测

打开最近一条 Trace，重点展示：

- 输入 / 输出
- 工具调用痕迹
- RAG 证据
- token 使用量
- 耗时

推荐说明：

> 这部分是为了让系统从“能回答”变成“可观测、可调优、可排障”。

### 6. 演示自动化任务中心

创建一个任务，例如：

- 类型：`knowledge_digest`
- 条件：仅在 `RAG 已连接` 时执行
- 调度：`Cron`

如果任务被阻塞，展示：

- 条件流判断
- 最近一次阻塞原因

如果任务能运行，展示：

- 运行记录
- Trace
- 渠道同步发送能力

### 7. 演示 MCP / Skills 扩展

在助手配置里展示：

- 可用 Skills
- 已加载 MCP
- MCP 校验与探测

推荐说明：

> 我把“模型能力”和“外部工具能力”分离了，方便不同助手装配不同工具集。

## 面试收尾说法

最后可以用这三句话总结：

1. 这是一个以控制台为中心的多助手 AI 应用系统，不是单功能机器人。
2. 我重点做了 RAG、任务调度、工具扩展、Trace 和工程化安全收口。
3. 这个项目已经具备继续往团队内部工具或业务助手方向演进的基础。
